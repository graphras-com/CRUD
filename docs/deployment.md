# Deployment

Docker image build, CI/CD pipelines, and Kubernetes deployment.

---

## Docker

### Dockerfile

The project uses a multi-stage Dockerfile:

**Stage 1 -- Frontend build:**
- Base: `node:22-alpine` (pinned to build host platform via `--platform=$BUILDPLATFORM` so npm never runs under QEMU)
- Runs `npm ci` and `npm run build`
- Auth env vars (`VITE_CLIENT_ID`, `VITE_TENANT_ID`, `VITE_API_SCOPE`) and build metadata (`VITE_BUILD_COMMIT`, `VITE_BUILD_TAG`, `VITE_BUILD_BRANCH`, `VITE_BUILD_TIME`) are baked in at build time via `ARG`

**Stage 2 -- Python runtime:**
- Base: `python:3.12-slim`
- Installs uv from `ghcr.io/astral-sh/uv:latest`
- Installs Python dependencies via `uv sync --frozen --no-dev`
- Copies `app/`, `resources/`, `base_data_import/`, `alembic.ini`, `alembic/`
- Copies built frontend from stage 1 into `static/`
- Creates `/data` directory for SQLite (used when `DATABASE_URL` is not set)
- Exposes port 8000
- Runs: `uv run uvicorn app.main:app --host 0.0.0.0 --port 8000`

### Docker Compose

```yaml
services:
  app:
    build:
      context: .
      args:
        VITE_CLIENT_ID: <spa-client-id>
        VITE_TENANT_ID: <tenant-id>
        VITE_API_SCOPE: <api-scope>
    ports:
      - "5173:8000"
    volumes:
      - app-data:/data
    env_file: .env
    environment:
      - DATABASE_PATH=/data/app.db
    restart: unless-stopped

volumes:
  app-data:
```

Key details:
- **Port mapping:** host port **5173** maps to container port 8000. Access at `http://localhost:5173`.
- **Data persistence:** SQLite database is stored in a named Docker volume (`app-data`) at `/data/app.db`.
- **Runtime config:** loaded from `.env` via `env_file`. Auth env vars (`AUTH_DISABLED`, `TENANT_ID`, etc.) are runtime. Vite env vars are build-time only.

### Building Manually

```bash
# Development (auth disabled)
docker compose up --build -d

# Production build with auth
docker build \
  --build-arg VITE_CLIENT_ID=your-client-id \
  --build-arg VITE_TENANT_ID=your-tenant-id \
  --build-arg VITE_API_SCOPE=api://your-api-id/access_as_user \
  -t crud-app:latest .
```

## CI -- GitHub Actions

The CI pipeline is defined in `.github/workflows/ci.yml`. It orchestrates reusable workflow files in the same directory.

### Pipeline Overview

```
push/PR to main
  |
  +-- detect-stack (auto-detect Python, frontend, Docker, E2E)
  |
  +-- py-lint        (ruff check + ruff format --check)
  +-- py-build       (pip install + optional compile check)
  +-- py-test        (pytest, matrix: Python 3.11/3.12/3.13, 85% coverage threshold)
  +-- fe-lint        (eslint)
  +-- fe-build       (vite build)
  +-- fe-unit        (vitest)
  +-- fe-e2e         (playwright, requires Python backend)
  +-- secrets-scan   (gitleaks)
  |
  +-- quality-gate   (all above must pass)
  |
  +-- docker-build   (PR only: build without push)
  +-- docker-push    (main/tags: build + push to ghcr.io)
  |
  +-- deploy-staging   (main only: deploy to staging K8s)
  +-- release          (tags only: creates GitHub release)
  +-- deploy-production (tags only: deploy to production K8s)
```

### Triggers

| Event          | What happens                                      |
|----------------|---------------------------------------------------|
| Push to `main` | Full CI + Docker push + deploy to staging       |
| Push tag `v*`  | Full CI + Docker push + GitHub release + deploy to production |
| Pull request   | Full CI + Docker build (no push)                   |
| Release        | Full CI + Docker push                              |
| Manual         | Full CI                                            |

### Docker Image

- **Registry:** `ghcr.io`
- **Image name:** `ghcr.io/<org>/<repo>` (lowercase, derived from `github.repository`)
- **Platforms:** `linux/amd64`, `linux/arm64`
- **Tags generated by `docker/metadata-action`:**
  - `main` branch: tagged as `staging` and `sha-<7chars>`
  - Semver tags: `v1.2.3` produces `1.2.3`, `1.2`, and `sha-<7chars>`
- **Cache:** GitHub Actions cache (`type=gha`)
- **Build args:** Vite auth env vars (from the GitHub Actions environment) and build metadata (`VITE_BUILD_COMMIT`, `VITE_BUILD_TAG`, `VITE_BUILD_BRANCH`, `VITE_BUILD_TIME`) are injected. The Docker build job receives the environment name (`staging` or `production`) from `ci.yml`, so `VITE_CLIENT_ID`, `VITE_TENANT_ID`, `VITE_API_SCOPE`, and `VITE_AUTHORITY` are read from environment-scoped variables.

### Configurable Variables

CI behavior can be tuned via GitHub repository variables (not secrets):

| Variable                       | Default          | Purpose                        |
|--------------------------------|------------------|--------------------------------|
| `CI_PYTHON_VERSION`            | `3.12`           | Python version for lint/build  |
| `CI_PYTHON_TEST_VERSIONS`      | `["3.11","3.12","3.13"]` | Test matrix             |
| `CI_PYTHON_INSTALL_SPEC`       | `.[dev]`         | pip install specifier          |
| `CI_PYTHON_COVERAGE_THRESHOLD` | `85`             | Minimum coverage percentage    |
| `CI_FRONTEND_NODE_VERSION`     | `22`             | Node.js version                |
| `CI_FRONTEND_WORKDIR`          | `frontend`       | Frontend directory             |

### Reusable Workflow Files

The `.github/workflows/` directory contains reusable workflows called by `ci.yml`:

| File                    | Purpose                              |
|-------------------------|--------------------------------------|
| `python-lint.yml`       | Ruff lint + format check             |
| `python-ci.yml`         | Install + optional byte-compile      |
| `python-tests.yml`      | pytest with matrix + coverage        |
| `frontend-lint.yml`     | ESLint                               |
| `frontend-build.yml`    | Vite production build                |
| `frontend-unit.yml`     | Vitest unit tests                    |
| `frontend-e2e.yml`      | Playwright E2E tests                 |
| `build-and-push.yml`    | Docker build + optional push         |
| `deploy-staging.yml`    | Deploy to staging Kubernetes         |
| `deploy-production.yml` | Deploy to production Kubernetes      |
| `python-gitleaks.yml`   | Secret scanning with gitleaks        |
| `python-release.yml`    | GitHub release creation              |

See `TEMPLATE_WORKFLOWS.md` and `.github/workflows/README.md` for detailed documentation of each reusable workflow.

## CD -- GitHub Actions

Continuous deployment to Kubernetes is handled by GitHub Actions workflows in `.github/workflows/`.

Deploy workflows use GitHub Actions [environments](https://docs.github.com/en/actions/deployment/targeting-different-environments/using-environments-for-deployment) (`staging` and `production`) for secret isolation and optional approval gates.

### Staging (`deploy-staging.yml`)

**Trigger:** automatically called by `ci.yml` after a successful Docker push on `main`. Can also be triggered manually via `workflow_dispatch`.

**What it does:**

1. Decodes `KUBECONFIG_B64` and optionally patches the API server URL.
2. Derives naming conventions from the repository:
   - `APP_NAME` from repo name (lowercased, dots/underscores replaced with hyphens)
   - `ORG_GHCR` from repo owner (for image registry path)
   - `ORG_DNS` from org name (e.g. `graphras-com` becomes `graphras.com`)
   - `NAMESPACE` = `{org}-staging`
   - `HOST` = `staging-{app}.{org-dns}`
3. Determines the image tag: `sha-<7chars>` from the source commit SHA.
4. Creates/updates all Kubernetes resources inline:
   - Namespace
   - GHCR image pull secret
   - CloudNativePG `Database` resource (in `default` namespace, shared `pg` cluster)
   - `DATABASE_URL` secret (assembled from the CNPG cluster's `pg-app` secret)
   - ConfigMap (auth env vars: `AUTH_DISABLED=false`, `TENANT_ID`, `API_AUDIENCE`, `CORS_ALLOWED_ORIGINS`)
   - Deployment (1 replica, startup/readiness/liveness probes on `/health`, resource limits)
   - Service (port 80 -> 8000)
   - Ingress (Traefik with TLS)
5. Waits for rollout (180s timeout).

### Production (`deploy-production.yml`)

**Trigger:** automatically called by `ci.yml` after a successful Docker push on a `v*` tag. Can also be triggered manually via `workflow_dispatch` with an explicit image tag.

**Differences from staging:**

| Aspect       | Staging                          | Production                    |
|--------------|----------------------------------|-------------------------------|
| Namespace    | `{org}-staging`                  | `{org}-prod`                  |
| Host         | `staging-{app}.{org-dns}`        | `{app}.{org-dns}`             |
| Image tag    | `sha-<7chars>` from commit SHA   | Semver from git tag (e.g. `1.2.3`) |
| DB name      | `{app}_staging`                  | `{app}_prod`                  |
| Environment  | `staging`                        | `production`                  |

### Required Secrets (GitHub Actions)

Deploy secrets should be configured in GitHub Actions [environments](https://docs.github.com/en/actions/deployment/targeting-different-environments/using-environments-for-deployment) (`staging` and `production`):

| Secret             | Purpose                                      |
|--------------------|----------------------------------------------|
| `KUBECONFIG_B64`   | Base64-encoded kubeconfig for k3s cluster     |
| `GHCR_USERNAME`    | GitHub username for pulling container images  |
| `GHCR_TOKEN`       | GitHub PAT with `read:packages` scope         |
| `K8S_API_SERVER`   | (Optional) Override k3s API server URL        |
| `TENANT_ID`        | Azure AD tenant GUID (for backend auth)       |
| `API_AUDIENCE`     | API app registration audience                 |

### Required Variables (GitHub Actions)

Frontend auth variables must be configured as **environment variables** in the `staging` and `production` environments (Settings > Environments > Variables). These are baked into the frontend at Docker build time:

| Variable           | Purpose                                                    |
|--------------------|------------------------------------------------------------|
| `VITE_CLIENT_ID`   | SPA app registration client ID from Entra                  |
| `VITE_TENANT_ID`   | Entra tenant ID (GUID)                                     |
| `VITE_API_SCOPE`   | Full API scope URI (e.g. `api://<api-id>/access_as_user`)  |
| `VITE_AUTHORITY`   | (Optional) Override MSAL authority URL                      |

Additional repository-level secrets:

| Secret              | Purpose                                    |
|---------------------|--------------------------------------------|
| `GITLEAKS_LICENSE`  | (Optional) Gitleaks license key             |

## Kubernetes Architecture

The application runs on **k3s** with the following components:

### Ingress

- **Ingress controller:** Traefik (default with k3s)
- **TLS:** auto-provisioned via TLS secret name convention (`{host}-tls` with dots replaced by hyphens)
- **Entry points:** `web` (HTTP) and `websecure` (HTTPS)

### Database

- **Operator:** CloudNativePG
- **Cluster:** a shared PostgreSQL cluster named `pg` in the `default` namespace
- **Per-environment databases:** created as `Database` CRDs (e.g. `{app}_staging`, `{app}_prod`)
- **Credentials:** the CNPG cluster's `pg-app` secret provides the username and password; the pipeline assembles the full `DATABASE_URL` and stores it in a per-namespace secret

### Application Pod

- **Container image:** pulled from `ghcr.io` via image pull secret
- **Configuration:** env vars from ConfigMap (`AUTH_DISABLED`, `TENANT_ID`, `API_AUDIENCE`, `CORS_ALLOWED_ORIGINS`) and secret (`DATABASE_URL`)
- **Health probes:**
  - Startup: `GET /health`, 5s initial delay, 5s period, 30 failures allowed
  - Readiness: `GET /health`, 10s period
  - Liveness: `GET /health`, 30s period
- **Resource limits:** 100m-500m CPU, 256Mi-512Mi memory

## Database Migrations

### Alembic (PostgreSQL)

Alembic migrations run automatically on startup for PostgreSQL deployments. The `app/main.py` lifespan handler detects `DATABASE_URL` and runs `alembic upgrade head`.

To generate a new migration after model changes:

```bash
uv run alembic revision --autogenerate -m "description of changes"
```

Rules:
- Never modify existing migration files in `alembic/versions/`
- Always create new revisions
- Test migrations locally before committing

### SQLite

SQLite deployments (local dev, Docker Compose) use `Base.metadata.create_all` on startup. No Alembic migrations are needed or applied.

## Release Process

1. Ensure all changes are on `main` and CI passes.
2. Create and push a semver tag:
   ```bash
   git tag v1.2.3
   git push origin v1.2.3
   ```
3. CI builds and pushes a Docker image tagged `1.2.3`, `1.2`, and `sha-<7chars>`.
4. CI creates a GitHub release.
5. CI deploys to production via the `deploy-production` workflow.
